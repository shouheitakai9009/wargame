# ディレクトリ構造

## やってはいけないこと

私は技術的な命名が嫌いです。例えば components/や types/や hooks/などといった命名です。
基本的に役割やドメインでディレクトリ構成は分けたいです。

## 基本ルール

- **ディレクトリ構成**: 役割の名前（例：`Header`）のディレクトリを作成し、その中に `index.tsx` を配置する構成とします。
- **コロケーション**: 関連するファイルは近くに置きます。例えば `Header` コンポーネントで使用するカスタムフックは `Header/useXxx.ts` のように配置します。

## 構造

### `src/`

- **`designs/`**: **ビジネスロジックを持たない**純粋な汎用 UI コンポーネントを配置します。ボタンや入力フォームなど、ドメインに依存しない見た目のみを管理します。
- **`widgets/`**: **ビジネスロジックを持つ**コンポーネントを配置します。
  - 階層化を推奨します（例: `Map/` の中にマスコンポーネントなどが含まれる）。
  - `widgets/` 直下には、比較的大きな単位のディレクトリ（機能の塊など）を配置します。
- **`states/`**: アプリケーションの状態管理や**ドメインの型・定数**を配置します。
  - 全てのドメインに対する型や定数はここで定義します（例: `soldier.ts` に兵種や兵の型・定数を定義）。
  - Redux の Slice などもここに含まれます。
- **`lib/`**: アプリケーション全体で使用する汎用的なユーティリティ関数などを配置します。
- **`routes/`**: ページ単位のコンポーネントを配置します。ルーティングに対応する画面の構成要素です。
- **`App.tsx`**: アプリケーションのルートコンポーネントです。
- **`main.tsx`**: アプリケーションのエントリーポイントです。
